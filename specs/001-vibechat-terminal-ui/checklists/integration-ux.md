# Integration & UX Requirements Checklist: VibeChat Terminal UI

**Purpose**: Валидация качества требований для интеграции хуков Claude Code и UX уведомлений Full Debug
**Created**: 2026-07-09
**Feature**: [spec.md](../spec.md)

**Note**: Чеклист сгенерирован `/speckit-checklist`. Проверяет качество требований, а не реализацию.

## Полнота требований (Requirement Completeness)

- [ ] CHK001 — Определены ли требования для каждого из 7 типов событий Full Debug (ToolUse, ThinkingBlock, SubagentSpawn, TokenCount, Timing, ModelSwitch, PermissionRequest)? [Completeness, Spec §FR-003]
- [ ] CHK002 — Определены ли требования к формату/структуре каждого типа уведомления? [Gap, Spec §Key Entities: AgentEvent]
- [ ] CHK003 — Задокументированы ли требования к поведению при запуске WebSocket-сервера (порт, хост, перезапуск)? [Gap, Spec §FR-002]
- [ ] CHK004 — Определены ли требования к хранению и восстановлению настроек фильтрации между сессиями? [Gap, Spec §FR-005, Assumptions]
- [ ] CHK005 — Описаны ли требования к поведению statusline в многострочном режиме (когда вывод терминала занимает несколько строк)? [Gap]
- [ ] CHK006 — Определены ли требования к взаимодействию statusline с другими терминальными UI элементами (строка ввода Claude Code, вывод ответа)? [Gap]
- [ ] CHK007 — Задокументированы ли требования к инициализации UI при старте сессии (пустой лог, начальное состояние statusline)? [Gap, Spec §US1]

## Ясность требований (Requirement Clarity)

- [ ] CHK008 — Является ли "Full Debug" однозначно определенным: перечислены ли все 7 типов событий с четкими границами? [Clarity, Spec §FR-003]
- [ ] CHK009 — Квантифицирован ли термин "плавное обновление statusline без мерцания" конкретными критериями (частота обновления, допустимое мерцание)? [Ambiguity, Spec §FR-008]
- [ ] CHK010 — Определен ли термин "gracefully обрабатывать отключения" для WebSocket конкретными параметрами (таймаут, кол-во попыток, стратегия backoff)? [Ambiguity, Spec §Edge Cases]
- [ ] CHK011 — Является ли "адаптивный дизайн для терминала" измеримым (минимальная ширина терминала, поведение при ресайзе)? [Ambiguity, Spec §FR-007]
- [ ] CHK012 — Определены ли критерии "удобной навигации" по логу (скорость скролла, поддержка клавиш PgUp/PgDn/стрелок, поиск)? [Ambiguity, Spec §US4-Scenario 2]

## Согласованность требований (Requirement Consistency)

- [ ] CHK013 — Согласованы ли списки типов событий Full Debug в FR-003, Key Entities: AgentEvent и User Story 1? [Consistency, Spec §FR-003 vs §Key Entities vs §US1]
- [ ] CHK014 — Согласованы ли требования к statusline (только текущее состояние) и логу (история всех событий) — не дублируют ли они функциональность? [Consistency, Spec §FR-003 vs §FR-004]
- [ ] CHK015 — Соответствует ли SC-004 (лог при >1000 записей) требованию SC-005 (WebSocket >100 событий/сек) — примерно 10 сек до заполнения лога, корректно ли это? [Consistency, Spec §SC-004 vs §SC-005]
- [ ] CHK016 — Соответствует ли требование "расширяемости для веб-GUI" (FR-010) протоколу WebSocket (FR-009) — одинаковые ли контракты событий? [Consistency, Spec §FR-009 vs §FR-010]

## Качество критериев приемки (Acceptance Criteria Quality)

- [ ] CHK017 — Может ли SC-001 (100% событий перехватываются) быть объективно измерен без доступа к исходному коду Claude Code? [Measurability, Spec §SC-001]
- [ ] CHK018 — Может ли SC-002 (statusline обновляется за <100 мс) быть верифицирован без специального инструментария? [Measurability, Spec §SC-002]
- [ ] CHK019 — Может ли SC-006 (повышение прозрачности на 80%) быть объективно измерен? Определена ли методология опроса? [Measurability, Spec §SC-006]
- [ ] CHK020 — Определены ли acceptance-критерии для сценария "переключение модели ИИ" (US1-Scenario 6) — какой именно вывод ожидается в statusline и логе? [Acceptance Criteria, Spec §US1-Scenario 6]

## Покрытие сценариев (Scenario Coverage)

- [ ] CHK021 — Определены ли требования для Alternate Flow: что отображается, когда агент не активен (idle состояние)? [Coverage, Gap]
- [ ] CHK022 — Определены ли требования для Exception Flow: что происходит при ошибке парсинга события от хука? [Coverage, Gap, Spec §Edge Cases]
- [ ] CHK023 — Определены ли требования для Recovery Flow: как UI восстанавливается после разрыва WebSocket соединения? [Coverage, Spec §Edge Cases]
- [ ] CHK024 — Определены ли требования для сценария "множественные одновременные субагенты"? [Coverage, Gap, Spec §US1-Scenario 3]
- [ ] CHK025 — Определены ли требования для сценария "Permission Request с таймаутом" — что отображается, если пользователь не ответил на запрос разрешения? [Coverage, Gap, Spec §US1-Scenario 4]

## Покрытие граничных случаев (Edge Case Coverage)

- [ ] CHK026 — Определены ли требования к поведению при недоступности порта для WebSocket-сервера (порт занят)? [Edge Case, Gap, Spec §Edge Cases]
- [ ] CHK027 — Определены ли требования к максимальному размеру лога (ротация, очистка, лимит памяти)? [Edge Case, Gap]
- [ ] CHK028 — Определены ли требования к поведению при одновременном изменении фильтров и активном потоке событий? [Edge Case, Gap, Spec §US2]
- [ ] CHK029 — Определены ли требования для терминалов без поддержки ANSI-кодов/псевдографики? [Edge Case, Gap, Spec §Assumptions]
- [ ] CHK030 — Задокументировано ли поведение при переполнении буфера событий (флуд >100 событий/сек)? [Edge Case, Spec §Edge Cases]

## Нефункциональные требования (Non-Functional Requirements)

- [ ] CHK031 — Определены ли требования к потреблению памяти логом при длительных сессиях (>1 час активной работы)? [NFR, Gap]
- [ ] CHK032 — Определены ли требования к влиянию хуков на время отклика Claude Code (оверхед < N мс)? [NFR, Gap]
- [ ] CHK033 — Определены ли требования безопасности для WebSocket-сервера (локальный only, аутентификация)? [NFR, Gap]

## Зависимости и предположения (Dependencies & Assumptions)

- [ ] CHK034 — Валидировано ли предположение "Claude Code предоставляет необходимые хуки для всех 7 типов событий"? Есть ли документальное подтверждение? [Assumption, Spec §Assumptions]
- [ ] CHK035 — Валидировано ли предположение "терминал поддерживает ANSI-коды" для целевой аудитории vibe-coders (Windows Terminal, Git Bash)? [Assumption, Spec §Assumptions]

## Примечания

- Все пункты проверяют качество требований, а не реализацию
- Отмеченные `[Gap]` указывают на потенциально отсутствующие требования
- Отмеченные `[Ambiguity]` указывают на неоднозначные формулировки
- Фокус: интеграция хуков + UX уведомлений, глубина: Standard
